    Exemplo Express:

        const express = require("express");
        const app = express();
        

        app.method(rota, middleware, controller) (ou req,res)

        app é o server express.
        method é o metódo http ("get", "post", put"...)

        Express também cria a função next() automaticamente então podemos estruturar o fluxo do nosso código de maneira
        simples.

    Vulnerabilidade na Query:


        *Código propositalmente vulnerável*


        URL: 

            /admin/users?user=joao

         middleware:

            auth(req, res, next){

                const user = req.query.user;   (?user=joao);

                if(!user){

                    res.end(...);

                 }else{

                 req.user = user;
                next();

                 }
                return;

                };

            *O middleware acima não autentica, ele apenas repassa o input*

         Por quê esse middleware é vulnerável? Simples, esse middleware não faz uma validação real da permissão do usuário,
         ele simplesmente pega o paramêtro user da query (que é controlado pelo cliente) e valida baseado nisso, ou seja,
         o sistema não realiza autenticação(verificação de identidade), ele trata input do cliente como fonte da verdade para autorização.
        Então qualquer um que fizer ?user=admin consegue acesso de admin ao site. 

         o controller faz isso:

             if(req.user!=="admin"){
            res.end(...);
            }
            ...


        Esse tipo de vulnerabilidade é conhecida como Broken Acess Control, pois o sistema confia em um paramêtro controlado pelo
        cliente para definir privilégios.

    Por quê esse tipo de vulnerabilide quebra o modelo de segurança?


        Violação de Trust-Boundary

        Simples, o server-side nunca deve confiar em dados enviados pelo client-side para autenticações ou verificações
        de autenticidade. Informações recebidas pelo client-side são altamente manipuláveis e caso sejam tratadas como "fonte da verdade"
        COM TODA CERTEZA, seu código estará vulnerável.

        Explicando melhor:

        Client-side: não confiável
        Server-side: fonte da verdade
        Caso atual: server terceira a verdade do cliente

    Implementei uma rota protegida (proteção simulada, ainda vulneravel por confiar no input do cliente),  /admin/users que 
    faz a verificação baseada em nome. E uma rota desprotegida /api/users, o objetivo é entender que um atacante consegue acessar
    os dados mesmo que uma rota esteja protegida, caso outra que acesse os mesmos dados não esteja.

    Exemplo:

    Usuário acessa http://localhost:3000/admin/users/?user= e o sistema realiza a verificação baseada em um
    identificador controlado pelo cliente (Inseguro!)

    Agora supondo que essa rota esta "protegida" o atacante consegue acesso ao array users, ja que há uma 
    rota desprotegida/endpoint sem controle de acesso

    que retorna os mesmos paramêtros. No caso:

    http:localhost:3000/api/users

    Essa rota não possui é totalmente aberta e consegue ser acessada sem nenhum tipo de validação

    Por quê essa falha é perigosa?

    1 - O sistema parece seguro?

    O sistema aparenta estar protegido ao analisar apenas o endpoint principal (/admin/users).
    No entanto, a existência de um endpoint secundário sem controle de acesso (/api/users) expõe
     o mesmo recurso, tornando a proteção ineficaz.

    2 - Por que desenvolvedores não percebem isso?

    Desenvolvedores frequentemente aplicam controle de acesso de forma pontual, assumindo que proteger endpoints críticos é suficiente.
    Porém, a falta de consistência na aplicação dessas regras permite que rotas secundárias exponham os mesmos dados sem proteção.


    3 - Qual impacto em produção?

    A falha permite acesso não autorizado a dados  sensíveis, podendo levar a exposição de informações internas, 
    escalonamento de privilégios e comprometimento da aplicação.

    4 - Como corrigir 

    A correção envolve garantir que o controle de acesso seja aplicado de forma consistente em todos os endpoints que expõem 
    dados sensíveis. Além disso, a identidade do usuário deve ser validada no servidor (ex: sessão ou token), sem confiar em dados
    fornecidos pelo cliente.
